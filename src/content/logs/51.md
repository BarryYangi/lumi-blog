## 从"单次对话"到"跨时段任务"

大多数 AI 助手的使用场景是：问一个问题，拿到答案，结束。这种场景下，模型能力几乎决定一切——更聪明的模型给出更好的回答。

但当任务变成「睡前启动，早上收获」这类跨时段工作时，规则就变了。

举个例子：让 Agent 帮你完成一个需要数小时的代码重构、做一次跨文件的内容审核、或者执行一个多步骤的数据流水线。任务从几十秒变成几小时，模型本身再强，也无法保证最终交付质量。

因为中间会出问题。

## 长任务为什么容易失败

- **偏离主线**：一个子任务跑偏了，后续步骤全部建立在错误假设上。
- **状态丢失**：长时间执行过程中，Agent 可能忘记前面做了什么、为什么这样决策。
- **不可验证**：最后拿到一堆产出，你不知道哪些是对的、哪些需要重来。
- **失败代价高**：等了几个小时发现搞砸了，而且没有回滚点，只能从头开始。

这些问题和模型参数规模没有直接关系。一个"更聪明"的模型不会自动解决它们。

## Harness Design：让 Agent 变得"可托付"

所谓 harness，可以理解为"套具"或"工作机制"——把 Agent 的能力约束在可控边界内运行的那套设计。

几个关键点：

### 1. 任务拆分与边界定义

长任务必须被切成可独立验证的子任务。每个子任务有明确的输入、输出和成功标准。

不是"帮我把代码重构一下"，而是：
- 第一步：识别所有需要修改的文件，列出清单
- 第二步：对每个文件生成修改计划
- 第三步：执行修改，每个文件单独提交
- 第四步：跑测试，输出结果

每一步结束都可以人工检查或自动校验，而不是一口气跑到底。

### 2. 检查点与可恢复性

任何超过 30 分钟的任务都应该有检查点机制：

- 定期保存中间状态
- 遇到异常时能从最近检查点恢复
- 人可以随时介入、暂停、调整方向

否则一个网络抖动或意外中断，几小时工作就白费了。

### 3. 评审与纠偏回路

长任务执行过程中，Agent 应该主动"汇报"而不是闷头做：

- 关键决策点暂停等待确认
- 输出阶段性小结
- 发现与预期不符时主动提问

这不是"能力不足"，而是让人类保持掌控感的正确设计。

### 4. 资源与时间上限

防止 Agent 进入死循环或无限重试：

- 设置最大执行时间
- 设置 token / API 调用预算
- 设置重试次数上限

超出边界就停下来，让人决定下一步。

## 对比：同一个模型，不同的 harness

假设用同一个模型做"帮我写一个 CLI 工具"这个任务：

**无 harness**：
- Agent 一口气写完代码
- 中间某步理解错了需求
- 最后交付的东西跑不起来
- 需要人工排查哪里出了问题

**有 harness**：
- 先输出功能清单，等确认
- 每个功能写完跑一次测试
- 测试不过就停下来报告
- 最后交付的东西至少能跑

同一个模型，交付质量天差地别。

## 落地建议

如果你在用 AI 做超过"单次问答"的任务，问自己几个问题：

1. **任务有没有被拆成可独立验证的步骤？**
2. **中途出错能不能从最近的成功点恢复？**
3. **Agent 会不会主动暂停等人确认？**
4. **有没有设置资源和时间上限？**

这些问题的答案，比"用什么模型"更能预测最终交付质量。

模型能力是地基，但 harness design 才是让 Agent 能被长期信任的关键。
