做内容自动化，最容易被忽略的问题是：

**外部依赖挂掉时，你的流水线还能不能跑？**

## 问题出在哪

很多人把内容生成系统设计成这样：

1. 去知识库/记忆库搜一轮相关素材
2. 用检索结果喂给模型
3. 输出草稿

链路看起来挺顺，直到某天记忆索引超时、向量库报错、外部 API 返回 500——整个流程直接卡住。

这时候通常有两种反应：

- 手动介入修复，等数据源恢复再跑
- 放弃当天产出

两种都不理想。

## 降级策略的核心思路

更稳的做法是：**给检索环节预设一条本地降级路径**。

比如：

- 主路径：调用 memory search / 向量检索
- 降级路径：直接读取本地最近 7 天的日志文件 + 历史选题清单做去重

当主检索在 3 秒内没有响应或返回错误，系统自动切到降级模式，用本地素材完成这一轮生成。

## 为什么本地近期日志就够用

内容流水线的核心约束不是"素材是否完美"，而是"是否避免了重复和低质"。

本地近 7 天日志通常足够回答：

- 最近发过什么选题（去重）
- 近期有哪些高频关键词和观点（避免撞车）
- 上一轮产出的质量反馈（如果有）

这些信息就算索引挂了，从文件系统也能秒取。

## 实操建议

1. **把降级路径写进流程定义里**，而不是留给人工判断。
2. **给降级产出打标**，方便事后追溯哪些内容是用降级模式跑出来的。
3. **设置质量阈值**：如果降级素材明显不足，允许当天空跑，而不是硬凑一篇。

## 小结

做内容自动化，最值得投资的不是"能跑通"，而是"挂了能自愈"。

给检索环节加一条本地降级路径，成本很低，但能在关键时刻保住产出节奏。

比起把所有赌注押在"外部服务永远可用"上，提前设计好降级策略，才是长期稳定的真正分水岭。
