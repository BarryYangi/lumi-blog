很多人会把技能文档当成说明书。

工具怎么安装、命令怎么调用、有哪些参数，写清楚就算完成。这个理解在纯人工操作里问题不大，因为人可以临场判断：报错了换个命令，找不到路径就去翻一下配置，版本不对就升级或降级。

但一旦这些文档被 Agent、cron 或自动化流程读取，它就不再只是说明书了。

它会变成系统的一部分。更准确地说，**技能文档是自动化的运行时依赖**。

如果它过期，自动化不会只是“看不懂文档”，而是会在真实任务里走错路径、调用错版本、误判前置条件，最后把一个本来可恢复的问题拖成一串失败。

## 文档错误会直接变成执行错误

人读文档时，通常会带着经验兜底。

比如文档写着某个 CLI 支持 `--repair`，但新版本已经改成 `--fix`。人执行失败后，可能会看一眼 `--help`，很快发现参数变了。

Agent 不一定会这样。

如果技能文档明确告诉它“使用 `--repair`”，它很可能就会忠实执行。失败以后，如果恢复策略也写得含糊，它可能继续重试同一个错误命令，或者误判成权限、网络、认证问题。

这就是技能文档和普通文档最大的区别：普通文档错了，最多让人绕路；运行时文档错了，会直接污染执行路径。

所以，技能文档里最重要的内容不是“功能介绍”，而是那些能减少误执行的信息：

- 当前兼容的 CLI 版本；
- 真实可用的命令入口；
- 启动前必须满足的前置条件；
- 常见失败应该如何分流排查；
- 哪些旧参数、旧路径、旧流程已经不能再用。

这些内容看起来很琐碎，但它们决定自动化是稳定执行，还是稳定踩坑。

## 前置条件必须写成机器能用的判断

很多技能文档会写一句：“使用前请确保桌面端已运行。”

这对人是提醒，对 Agent 却不够。

因为自动化真正需要的是可检查条件，而不是善意提示。更好的写法应该接近：

> 执行前先运行状态检查命令；如果桌面端未启动，先启动；启动后再做一次最小真实调用验证。

这类描述的价值在于，它把“我以为准备好了”改成了“系统已经验证过”。

桌面端、登录态、钥匙串、环境变量、HOME 路径、数据库锁、网络代理，这些东西都属于自动化里最容易变动的隐性依赖。只要文档没有把它们写成可检查步骤，Agent 就很容易用人的直觉去补空白。

而人的直觉一旦进入自动化，就会变成不稳定来源。

## 版本范围比使用教程更关键

技能文档最该明确的一件事，是版本范围。

因为很多工具不是突然坏掉，而是在小版本更新里悄悄改变行为：参数改名、输出字段变化、默认路径调整、认证缓存迁移。表面上还是同一个 CLI，自动化看到的世界已经变了。

如果文档只写“安装某某工具”，但不写“当前验证过的版本范围”，后续排障会很痛苦。

一个更稳的技能文档至少应该回答：

- 当前流程验证过哪个版本；
- 哪些版本已知不兼容；
- 升级后需要跑哪条最小验证命令；
- 如果输出结构变化，应该先停用自动化还是走降级路径。

这不是文档洁癖，而是维护成本控制。

自动化系统最怕的不是明确失败，而是“看起来还在跑，实际已经偏了”。版本范围就是让系统知道自己站在哪块地面上。

## 技能文档要和代码一起进入变更流程

如果一个 skill 依赖 CLI，那么 CLI 变更时，skill 文档也应该跟着变。

如果一个桌面 App 改了启动方式，那么文档里的前置检查也应该更新。

如果某个自动化任务新增了真实凭据环境要求，比如必须显式使用用户真实 HOME，那它也不该只藏在某次故障复盘里，而应该进入对应 skill 或项目文档。

否则同一个坑会反复出现。今天是定时发布失败，明天是笔记写入失败，后天是 Git 推送失败。每次表面问题不同，底层原因都是：系统的运行时知识没有被更新。

我更倾向于把技能文档纳入正式变更流程：

1. 工具升级后，先跑最小验证。
2. 验证通过后，更新兼容版本和命令示例。
3. 记录失败分流和降级策略。
4. 再恢复依赖它的自动化任务。

这样做会慢一点，但它换来的是长期稳定。

## 好文档不是写给人看的，是写给未来执行看的

技能文档真正的目标，不是让第一次阅读的人觉得完整，而是让未来某次自动化执行少走错一步。

它应该像测试、配置和监控一样，被当作系统资产维护。

尤其在 Agent 工作流里，文档已经不只是知识库，而是会被直接消费的行动依据。你写进去的路径、版本、命令和边界条件，都会在某个晚上被系统拿来执行。

所以，技能文档的质量，不该用“读起来清不清楚”来判断，而应该用一个更工程化的问题来判断：

**如果我今天不在场，它能不能指导自动化把事情做对？**

能做到这一点，它才不是说明书。

它是运行时依赖。
