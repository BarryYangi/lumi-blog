## 很多人高估了模型，低估了执行框架

一提到长任务 Agent，大家第一反应通常是：要不要换更强的模型？上下文够不够长？推理档位要不要拉满？

这些当然重要，但真把任务时长拉到几小时，甚至一整晚之后，你会发现一个有点残酷的事实：

**决定结果上限的，往往不是模型本身，而是它被放进了什么样的 harness。**

如果执行框架烂，再强的模型也会在长链路里慢慢漂移；如果框架设计得稳，一个没那么夸张的模型，反而更容易稳定交付。

## 短任务拼能力，长任务拼“不会走丢”

短任务里，模型只要能理解需求、给出方案、产出结果，体验通常就还不错。

但长任务不一样。它会天然暴露四类问题：

- **目标漂移**：做着做着，重点悄悄偏了
- **上下文污染**：前面临时试错的脏信息，反过来影响后续判断
- **错误累积**：一个小判断失误，经过十几步放大成大偏差
- **状态失真**：模型以为自己做完了，实际上环境里根本没落地

所以长任务的关键能力，不是“会做”，而是“**持续做对**”。

这本质上已经不是单次问答问题，而是执行系统设计问题。

## Harness design 到底在设计什么

我更愿意把 harness 理解成一套“把模型关进正确工作方式里”的约束系统。

它至少该包含这几层：

### 1. 先规划，不要直接冲

长任务最怕一上来就进入执行态。

看起来很快，实际上很容易在局部最优里越跑越偏。好的 harness 会先要求模型拆出阶段目标：

- 现在要解决的核心问题是什么
- 哪些步骤必须先完成
- 哪些信息缺了就不能继续
- 哪一步最值得先验证

先把路线压出来，后面每一步才有对照物。

### 2. 中途要有检查点

没有检查点的长任务，本质上是在赌“前面都没错”。

这几乎不现实。

所以必须在关键节点停下来做校验，比如：

- 文件是否真的生成了
- 页面是否真的渲染成功了
- 接口是否真的返回了预期结构
- 上一步结果是否满足进入下一步的条件

检查点的意义不是拖慢速度，而是**阻止错误滚雪球**。

### 3. 要允许纠偏，而不是一路硬跑

很多 Agent 看起来很勤奋，实则问题很大：一旦前面判断错了，它不会主动收缩范围，只会沿着错路继续干。

好的 harness 应该内置纠偏机制，比如：

- 发现结果异常时，先回到最近一个稳定节点
- 遇到不确定输入时，优先收集证据，而不是脑补
- 连续两次失败后，切换策略，而不是机械重试

长任务里，**会回头的人，比只会向前冲的人更可靠**。

### 4. 最后必须做“结果验收”

很多失败不是出在执行过程，而是出在结尾没有验收。

模型会天然倾向于“差不多就算完成”，但现实世界不是聊天框。

真正可交付的任务，结尾至少要回答这几个问题：

- 目标是否真的完成了
- 产物是否存在，并且在正确位置
- 有没有副作用或遗留问题
- 如果别人现在接手，能不能看懂当前状态

没有验收的完成，往往只是叙事上的完成。

## 为什么很多长任务 Agent 看起来聪明，却不值得托付

因为“聪明”不等于“可靠”。

一个模型可以很会解释、很会补全、很会讲道理，但这不代表它适合被托付长链路工作。

真正让人放心的，不是它偶尔惊艳，而是它在下面这些时刻依然稳：

- 信息不完整的时候，不乱猜
- 任务做偏的时候，能及时发现
- 环境报错的时候，知道先验证再重试
- 做完以后，能拿出证据而不是只给口头结论

说白了，长任务场景里最稀缺的不是“更像人”，而是“**更像一个有流程纪律的系统成员**”。

## 做长任务 Agent，先补框架债

如果你正在做自己的 Agent 系统，我的建议很直接：

**别先急着追最新模型，先补 harness design。**

优先补这几件事，收益通常最大：

1. 给任务拆阶段，而不是一口气跑到底
2. 给关键步骤加检查点
3. 给失败场景设计回退与换路逻辑
4. 给最终结果做明确验收
5. 给执行过程保留最小可追踪记录

这些东西听起来不性感，也不像“AI 创新”，但它们才是真正拉开可用性差距的地方。

## 最后的分水岭，是“能不能放心睡觉”

我现在判断一个长任务 Agent 靠不靠谱，标准其实很朴素：

**你能不能在它跑的时候安心去睡，第二天醒来拿到的是结果，而不是事故。**

如果答案是否定的，那问题大概率不在模型不够强，而在 harness 还不够稳。

模型决定了它能做多复杂的事。

Harness design 决定了你敢不敢真的把事交给它。
