想把家里的 NAS、公司的开发机、VPS 上的服务串成一个内网，是很多开发者都折腾过的事。Tailscale、ZeroTier、WireGuard、frp……工具选择一大把。

但很多人一上来就装工具、配密钥、改防火墙，折腾半天发现设备之间就是连不通。然后开始怀疑工具有 bug、配置写错、端口被封。

其实大部分情况下，问题不出在工具，而出在 NAT 类型。

## NAT 是什么？为什么重要？

简单说，NAT（网络地址转换）是路由器用来把内网设备"藏"在一个公网 IP 后面的技术。你家里所有设备可能共享同一个公网 IP，路由器负责把流量分发给正确的设备。

问题在于：NAT 的实现方式有好几种，有的对打洞（hole punching）友好，有的几乎不可穿透。

常见分类：

- **Full Cone**：最宽松，任何外部主机都能向你发数据，打洞最容易。
- **Restricted Cone / Port Restricted Cone**：有限制，但打洞通常能成功。
- **Symmetric NAT**：每次连接到不同目标时端口都会变，打洞几乎不可能。

如果你的设备在 Symmetric NAT 后面，那不管用什么 P2P 工具，直连都会失败，只能走中继（relay）。

## 怎么测？

最简单的方法是用 STUN 测试工具：

```bash
# 用 pystun3（Python）
pip install pystun3
pystun3

# 或者用 stun-client
stun stun.l.google.com:19302
```

输出会告诉你 NAT 类型，比如 `Full Cone`、`Symmetric` 之类。

Tailscale 也有内置检测：

```bash
tailscale netcheck
```

会直接告诉你当前网络环境是否支持直连，以及 DERP 中继延迟。

## 知道 NAT 类型后怎么办？

**如果是 Full Cone 或 Restricted Cone：** 恭喜，大多数 P2P 组网工具都能直连成功。配好密钥和防火墙就行。

**如果是 Symmetric NAT：**

1. 检查路由器设置，有些路由器可以切换 NAT 类型（尤其是一些企业级或可刷固件的路由器）。
2. 如果改不了，就接受现实：直连走不通，只能用中继。Tailscale 的 DERP、ZeroTier 的 relay、自建 frp 都是方案。
3. 也可以考虑在一端用公网 IP 或有公网 IP 的 VPS 做"锚点"，让其他设备主动连过去。

## 为什么要先测再折腾？

因为 NAT 类型是网络拓扑层面的约束，不是软件配置能改变的。

如果你不先搞清楚这一层，可能会在配置文件、端口映射、防火墙规则上浪费几个小时，最后发现问题根本不在那里。

先花两分钟跑个 STUN 测试，知道自己的网络环境是"可穿透"还是"必须中继"，后续排障路径会清晰很多。

有时候，问题不是工具不行，而是物理定律（好吧，是网络协议）不允许。先认清边界，再找解法。
