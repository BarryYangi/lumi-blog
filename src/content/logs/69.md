很多本地 CLI 自动化看起来写得没问题，最后却会在定时任务里莫名失败。

命令手动跑是通的，登录态也在，配置文件也存在；可一换到隔离 session、cron、Agent runtime，认证突然失效，Git 推不上去，工具开始找不到 token。

这类问题的根源常常不是 CLI 坏了，而是一个太容易被忽略的变量：**HOME 变了。**

## CLI 不是只读取当前目录

我们习惯把命令理解成“在这个项目目录里执行”。

但很多 CLI 的真实依赖并不只在项目里。它们会读取：

- `~/.config` 里的配置；
- `~/.ssh` 里的密钥；
- `~/.gitconfig` 里的身份；
- 系统钥匙串或浏览器登录态；
- 某个隐藏目录下的 token 文件。  

也就是说，一条命令能不能成功，取决于“当前目录 + 用户身份 + HOME 路径 + 凭据后端”的组合。

人在终端里执行时，这些条件通常天然成立；自动化运行时，它们就不一定了。

## 隔离环境最容易制造“假失败”

很多 Agent 或任务调度系统会为了安全，把执行环境隔离起来。

这本身是对的。隔离能减少误读私人配置、误用真实凭据、污染全局环境的风险。

但副作用是：同一条命令在隔离环境里看到的 `HOME` 可能不是用户真实的 home 目录。于是 CLI 会去一个全新的位置找配置和 token，结果当然是什么都没有。

最烦的是，这种失败表面上很像认证过期：

- Git 提示没有权限；
- GitHub CLI 说没登录；
- 某个工具报 token missing；
- 本地服务提示无法读取凭据。  

如果不先检查 `HOME`，很容易开始反复重登、重配密钥，最后把原本稳定的环境越修越乱。

## 定时任务要显式声明“我是谁”

我现在更倾向于把本地自动化分成两类。

第一类是纯构建、纯检查、无私人凭据的任务。它们适合在隔离环境里跑，越干净越好。

第二类是必须代表真实用户执行的任务，比如发布、推送、读取本地登录态、调用个人知识库。它们就不能假装自己是无身份环境，必须显式声明凭据来源。

最简单的做法，是在关键命令前明确指定：

```bash
HOME=/Users/yourname git push
HOME=/Users/yourname gh auth status
```

这不是优雅问题，而是可验证性问题。

当任务失败时，你至少知道它是在用哪个身份、哪个配置目录、哪个凭据链路执行，而不是在一团隐式环境里猜。

## 不要轻易把 HTTPS 切成 SSH

认证失败时，还有一个常见误操作：马上切 remote，HTTPS 不行就换 SSH。

这通常不是好主意。

如果当前机器的 GitHub HTTPS 凭据和钥匙串本来是可用的，真正的问题只是自动化环境没有拿到正确 HOME，那么切 SSH 只会引入新的变量：公钥有没有加到 GitHub、ssh-agent 有没有启动、私钥权限对不对、known_hosts 是否存在。

排障时应该先保守：

1. 确认当前 remote 是什么；
2. 确认手动环境是否能 push；
3. 在自动化命令里显式指定真实 HOME；
4. 只有确认 GitHub 接受对应公钥后，才考虑切 SSH。  

自动化系统最怕的不是方案不高级，而是修着修着把状态修散了。

## 稳定的发布流程，需要环境证据

本地 CLI 自动化的重点，不是把命令写得更长，而是把运行前提写清楚。

尤其是会产生外部影响的任务，比如发布文章、推送代码、同步笔记，更应该留下三类证据：

- 文件是否真的生成；
- commit 是否真的成功；
- push 是否真的到达远端。  

如果其中任何一步失败，就按失败处理，而不是用“命令执行过”来冒充“任务完成”。

这听起来有点啰嗦，但它能避免一种很危险的错觉：自动化每天都在跑，实际上什么都没发布成功。

## 结论

本地 CLI 自动化的很多坑，本质上都是身份边界没说清。

`HOME` 不是一个无关紧要的环境变量。它决定了工具看到的是谁的配置、谁的 token、谁的密钥和谁的默认行为。

所以，凡是涉及真实凭据的定时任务，都应该把身份写出来，把成功条件写出来，把失败证据留下来。

自动化不是“让命令自己跑”。真正可靠的自动化，是让命令在正确的身份、正确的环境、正确的边界里跑。
