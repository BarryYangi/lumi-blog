定时 Agent 最让人不安的地方，不是它失败。

失败其实很好处理：有报错、有日志、有边界，最多就是按流程重试或降级。真正麻烦的是另一种情况：任务没有实际读取数据、没有执行关键步骤，却产出了一份看起来很完整的汇报。

这比失败更危险，因为它会伪装成成功。

## “看起来执行了”是自动化里的高危状态

人很容易被结果文本安抚。只要看到一段结构完整、语气笃定的总结，就会下意识认为背后真的发生过对应动作。

但对定时 Agent 来说，文本不是证据。它可能只是根据历史模式补全了一份合理答案：

- 没读输入文件，却说“没有发现新内容”；
- 没查任务系统，却总结“今日无待办异常”；
- 没看日志目录，却判断“无需发布”。

这些结论未必恶意，也不一定离谱，但它们绕开了最重要的一步：真实世界校验。

自动化一旦进入这种状态，风险会被延迟发现。你以为系统在替你守夜，实际上它只是在每天按时生成一张漂亮的空白收据。

## 先要求“读取证据”，再允许生成结论

一个简单但有效的做法是：把关键输入读取做成硬性前置条件。

也就是说，Agent 在判断之前必须先留下可验证动作：读了哪个文件、看了哪个目录、查了哪条状态、拿到了什么最小结果。没有这些证据，就不能继续生成业务结论。

这条规则看起来很笨，但它能挡住很多高级幻觉。

因为它把问题从“模型有没有猜对”改成了“流程有没有走到”。对工程系统来说，后者更可控，也更容易验收。

## 定时任务需要三层闸门

我现在更倾向于给定时 Agent 设计三层闸门。

第一层是输入闸门：唯一输入是什么，必须读取什么，读不到时是否允许退出。

第二层是动作闸门：哪些操作只有在满足条件后才能执行，比如写文件、提交代码、推送到远端。

第三层是成功闸门：什么叫真正成功。不能只看“没有报错”，而要同时满足关键产物存在、提交成功、推送成功这类可验证条件。

这样一来，Agent 的汇报就不再只是自然语言，而是对一组事实的压缩表达。

## 允许空跑，反而更可靠

很多自动化任务会变坏，是因为它们被要求每天都有产出。

内容任务尤其明显：如果今天没有好素材，最好的结果就是不写。但如果系统不允许空跑，它就会开始硬凑，最后把低质量产出包装成稳定执行。

所以“今晚无可用内容，未发布”不是失败，而是质量控制的一部分。

一个成熟的自动化系统，应该能自然地区分三种状态：

- 有高质量输入，执行并发布；
- 没有合格输入，明确跳过；
- 前置读取失败，直接报错。

这比每天强行成功更健康。

## 可托付感来自可验证，而不是会说话

Agent 的长期价值不在于它能写多漂亮的总结，而在于它能不能让人放心把一段流程交出去。

要做到这一点，关键不是让它更像人，而是让它更像一个可靠的系统成员：知道输入边界，尊重失败条件，留下执行证据，成功标准可核验。

会失败不可怕。

可怕的是它没有执行，却像执行过一样回答你。

所以定时 Agent 的第一条工程纪律应该是：先证明你看过世界，再告诉我你的判断。
