"睡前给 Agent 一个任务，早上起来收结果"——这个场景听起来很美好。但真正跑起来，你会发现长任务自动化的坑比想象中多得多。

## 核心挑战：无人值守时的不确定性

短任务失败了，重跑一次就好。但长任务不一样：

- 网络抖动、API 超时、鉴权过期，任何一个都可能在凌晨 3 点悄悄发生
- 你醒来看到的不是结果，而是一堆重试失败的日志
- 更糟的是：任务跑到一半崩了，前面的进度全丢

所以，长任务自动化的设计目标不是"不出错"，而是**出错时能恢复、能止损、能让你知道发生了什么**。

## 四个必须做的设计

### 1. 任务边界要明确

"帮我处理这批数据"不是一个好的长任务定义。

好的定义是：
- 输入是什么、放在哪
- 输出是什么、写到哪
- 成功的判定标准是什么
- 失败时应该怎么处理

边界清晰了，Agent 才知道什么时候该停、什么时候算完成。

### 2. 检查点是必需品，不是可选项

长任务最怕的不是失败，而是"从头再来"。

每完成一个阶段，把状态写下来：
- 已处理到哪一条
- 中间结果存在哪
- 下次从哪里继续

这不是过度设计，而是基本保障。研究显示，长时间运行的任务失败率可以高达 30%，而检查点机制能挽回超过 60% 的重复计算。

### 3. 失败恢复要前置设计

很多人的失败处理是"出了问题再说"。但长任务没有这个奢侈——你睡着了，没法"再说"。

需要提前想好的：
- 鉴权失效怎么办（是暂停等待，还是切换备用凭证）
- 网络超时重试几次、间隔多久
- 连续失败达到什么阈值要停下来并通知

把这些写成代码里的默认行为，而不是临场判断。

### 4. 资源上限必须有

Agent 自己跑一晚上，最怕的是失控：
- 陷入死循环，疯狂重试
- API 调用超出预算
- 磁盘被日志或中间文件塞满

设定硬限制：最大运行时长、最大 API 调用次数、最大存储占用。触及上限就停下来，哪怕任务没完成。

**保护系统比完成任务更重要。**

## 一个实用的检查清单

在让 Agent 跑长任务之前，过一遍这个清单：

- [ ] 输入/输出/成功标准都写清楚了吗？
- [ ] 有检查点机制吗？崩溃后能从中间恢复吗？
- [ ] 鉴权失效有处理方案吗？
- [ ] 重试策略配好了吗（次数、间隔、退避）？
- [ ] 资源上限设了吗？
- [ ] 异常时会通知我吗？

全部打勾再睡觉，早上才能真的收到结果，而不是一堆待处理的故障。

---

长任务自动化的本质，是把"你在旁边盯着"时的各种临场判断，提前固化成系统行为。

做好这件事，Agent 才能从"需要看着"变成"真正自主"。
