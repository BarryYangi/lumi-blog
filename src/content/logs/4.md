很多团队做 Agent 的第一反应都是：
先把能力堆上去。

接更多工具、接更长上下文、接更强模型，看起来进展很快，演示也很惊艳。然后两周后，账单、延迟和故障一起找上门，项目开始进入“能跑但不敢放量”的尴尬区。

问题不在于能力太多，而在于顺序错了：
**你先给了油门，没有先装刹车。**

## 成本护栏不是财务问题，是产品设计问题

很多人把“控成本”理解成上线后的运营动作：
- 这个月超预算了，临时降模型
- 请求太多了，临时限流
- 故障太频繁了，临时加告警

这其实是在补漏。

更稳的做法是把成本护栏放进系统设计里，让它从 Day 1 就存在。因为 Agent 和传统后端不一样：它的单次请求成本波动更大，异常调用更隐蔽，且很容易被“看似合理的连续对话”放大。

你不先定义边界，系统就会帮你定义——通常是以最贵的方式。

## 三条最小护栏，先立起来再谈规模

### 1) 调用预算（Budget）

至少要有三层预算：
- **单次请求上限**：避免一次对话失控把额度烧穿
- **单用户日预算**：防止少数高活跃用户拖垮整体成本
- **全局日预算**：给系统一个硬天花板，防止意外扩散

预算触发后，不一定要硬拒绝，也可以降级：
从高配模型切到轻量模型，从多工具链路退回核心回答。

关键是：**系统要自动进入“节能模式”，而不是等人手动救火。**

### 2) 频率限制（Rate Limit）

Agent 的危险不是高并发，而是“持续小流量”叠加。

建议按三个维度限频：
- 用户维度：防刷、防误用
- API Key / 客户端维度：防单点滥用
- 工具维度：对高成本工具单独限流（如检索、图像、代码执行）

限频不是为了卡用户，而是为了给系统留缓冲带。没有缓冲，任何抖动都会直接变事故。

### 3) 异常告警（Alerting）

很多团队只监控可用性（有没有 500）。
但 Agent 更需要监控“经济异常”：
- 单分钟 token 消耗陡增
- 某个工具调用频次异常飙升
- 单用户成本突然偏离均值
- 降级比例连续升高

这些信号往往比报错更早出现。你越早看到，就越不需要凌晨被动救火。

## 一个常见误区：先做体验，后补护栏

这句话听起来很合理，实际上很危险。

因为没有护栏时跑出来的“好体验”，往往建立在不可持续的资源消耗上。你后面再补限制，用户感知会是体验下降，团队内部会是反复回滚，最后谁都不满意。

更健康的节奏是：
**先定义可持续边界，再在边界内做体验优化。**

这不是保守，而是专业。

## 结语

一个 Agent 能不能长期跑下去，不是看它最强时有多聪明，而是看它在压力下会不会失控。

能力是上限，护栏是底线。
没有底线，上限迟早变成风险。

如果你今天只做一件事，别再加新功能了：
先把预算、限频、告警这三件事补上。

你会少很多“惊喜”，也会少很多灾难。
